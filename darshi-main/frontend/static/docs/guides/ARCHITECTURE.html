Architecture Layers

1. Presentation Layer (Frontend)

  • SvelteKit 5 with Runes API (state,derived, effect,props)
  • Static Site Generation (SSG) on Cloudflare Pages
  • Progressive Web App (PWA) capabilities
  • Service Worker for offline support and push notifications
  • Responsive design (mobile-first)
  • Dark mode support (Uber Black theme)
  • Bilingual i18n (English and Hindi)

2. API Gateway Layer (Nginx + FastAPI)

  • Nginx reverse proxy with SSL termination
  • FastAPI application with async support (Python 3.11)
  • JWT authentication and authorization
  • OAuth 2.0 integration (Google, GitHub, Facebook)
  • Rate limiting (tiered: anonymous 3/hr, registered 10/hr, admin unlimited)
  • Request validation and sanitization
  • Security headers (CSP, HSTS, X-Frame-Options)

3. Business Logic Layer (Services)

  • Modular service architecture (service functions, no classes)
  • Circuit breakers for external APIs (Gemini, Nominatim)
  • Retry logic with exponential backoff
  • Caching strategies (Redis with in-memory fallback)
  • Background task processing (async tasks)
  • Image optimization (WebP conversion)
  • Perceptual hashing (dHash) for duplicate detection

4. Data Layer

  • PostgreSQL 15 with PostGIS (relational database with geospatial extensions)
  • Redis 7 with AOF persistence (caching and rate limiting)
  • Cloudflare R2 (S3-compatible object storage for images)
  • Docker volumes for data persistence

5. External Integration Layer

  • Google Gemini 2.5 Flash (AI image analysis and verification)
  • Nominatim (geocoding with India filter)
  • Overpass API (nearby landmarks)
  • OAuth providers (Google, GitHub, Facebook)

Data Models

PostgreSQL Tables

1. users Table

Fields:

  • id - Auto-incrementing primary key
  • username - Unique username (case-sensitive)
  • email - Optional email (not unique, multiple accounts can use same email)
  • phone - Optional phone number (E.164 format)
  • password_hash - Bcrypt hash (null for OAuth users)
  • role - User role: citizen, moderator, super_admin
  • is_active - Account active status
  • is_verified - Email verification status
  • oauth_provider - OAuth provider name
  • oauth_id - OAuth provider user ID
  • created_at - Account creation timestamp
  • updated_at - Last update timestamp

2. reports Table

Report Status Enum:

  • PENDING_VERIFICATION - Awaiting AI verification
  • VERIFIED - AI verified as valid civic issue
  • REJECTED - AI rejected (not a civic issue)
  • DUPLICATE - Duplicate of existing report
  • IN_PROGRESS - Admin marked as being worked on
  • RESOLVED - Admin marked as resolved
  • FLAGGED - System error during processing

Key Fields:

  • report_id - UUID for public identification
  • geom - PostGIS geography point for spatial queries
  • geohash - Geohash for proximity queries (precision 7 = ~153m × 153m grid)
  • image_data - JSONB with WebP and JPEG URLs
  • timeline - JSONB array of state change events
  • dhash - Perceptual hash for duplicate detection
  • duplicate_of - Foreign key to original report if duplicate

Service Layer Design

Error Handling Architecture (3-Layer Defense)

Layer 1: Prevention

  • Input validation at router layer (core/validation.py)
  • Type checking (Pydantic models)
  • Sanitization (sanitize_form_data())

Layer 2: Recovery

  • Retry decorators: @retry_database_operation, @retry_ai_operation, @retry_storage_operation
  • Circuit breakers: Gemini (5 failures/120s), Nominatim (3 failures/60s)
  • Graceful degradation (continue with partial data)

Layer 3: Communication

  • Structured error responses with request IDs
  • User-friendly messages via toast notifications
  • Detailed logging with context (exc_info=True)

    • HTML entity encoding
    • SQL injection prevention via ORM

  2. Rate Limiting:

    • Anonymous: 3 requests/hour for report submission
    • Registered: 10 requests/hour for report submission
    • Admin: Unlimited
    • Stored in Redis with sliding window

  3. CORS Configuration:

  4. Security Headers:

    X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Content-Security-Policy: default-src 'self'; ...

Design Decisions

Why PostgreSQL over Firestore?

Reasons for Migration:

  1. Relational Data Model: Reports, users, comments, upvotes have clear relationships
  2. ACID Transactions: Need for atomic updates (upvote count, comment count)
  3. Geospatial Queries: PostGIS provides powerful spatial functions
  4. Cost Predictability: Fixed VPS cost vs. Firestore usage-based pricing
  5. Better Tooling: pgAdmin, pg_dump, familiar SQL ecosystem
  6. Migrations: Alembic provides version-controlled schema changes

Trade-offs:

  • Lost: Auto-scaling (Cloud Run scaled to zero when idle)
  • Gained: Full control, better performance, predictable costs

Why Cloudflare R2 over Google Cloud Storage?

  1. No Egress Fees: GCS charges for bandwidth, R2 does not
  2. S3 Compatibility: Standard boto3 API
  3. Global CDN: Images served from edge locations
  4. Cost: ~70% cheaper than GCS

Why Docker Compose over Kubernetes?

  1. Simplicity: Single VPS, no need for orchestration complexity
  2. Resource Efficiency: Kubernetes overhead not justified for small scale
  3. Easy Management: Docker Compose is simpler to debug and maintain
  4. Sufficient for Scale: Current traffic fits well on single 16GB VPS

Why SvelteKit 5 over React/Next.js?

  1. Performance: Svelte compiles to vanilla JS (no virtual DOM)
  2. Bundle Size: Svelte apps are ~30-40% smaller than React equivalents
  3. Developer Experience: Less boilerplate, more intuitive reactivity
  4. Runes API: Modern reactivity primitives (state,derived, $effect)
  5. SSR/SSG: Built-in support for server-side rendering

Why Module-Level Functions over Classes?

Backend Services:

  • Simpler to test (no state)
  • No constructor complexity
  • No self parameter noise
  • Easier to mock in tests
  • Follows functional programming principles

Why Redis for Rate Limiting?

  1. Atomic Operations: INCR and EXPIRE are atomic
  2. Sliding Window: Can implement complex rate limit algorithms
  3. Fast: In-memory, sub-millisecond latency
  4. Persistence: AOF ensures rate limit state survives restarts
  5. Shared State: Multiple backend instances can share rate limit counters
