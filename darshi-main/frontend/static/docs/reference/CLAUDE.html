Project Overview

Darshi is a civic grievance platform that allows citizens to report and track civic issues (potholes, garbage, streetlights, etc.) using AI-powered image analysis. The system uses Google Cloud services for storage, database, and AI capabilities.

Project Structure:

  • / - FastAPI backend (Python)
  • /frontend - SvelteKit frontend (TypeScript)

Core Architecture

Backend Stack:

  • FastAPI web framework with async support
  • Google Cloud Firestore as the primary database
  • Google Cloud Storage for image uploads
  • Google Gemini (gemini-1.5-flash) for AI-powered image analysis
  • BigQuery for analytics and hotspot detection
  • JWT authentication with bcrypt password hashing

Frontend Stack:

  • SvelteKit 5 with Runes (reactive programming)
  • TypeScript for type safety
  • Native CSS (no frameworks - ultra lightweight)
  • Vite for fast development and building

Application Flow:

  1. User submits report with images via /api/v1/report
  2. System calculates SHA256 hash for duplicate detection
  3. Images uploaded to GCS bucket {PROJECT_ID}-reports
  4. Background task processes AI verification:
    • Downloads image from GCS
    • Analyzes with Gemini to classify civic issues
    • Checks for duplicate reports by geohash and category
    • Updates report status (VERIFIED, REJECTED, DUPLICATE)
  5. Timeline events tracked for audit trail
  6. BigQuery analytics identify hotspots (3+ severe issues in 24h)

Service Layer Structure:

  • ai_service.py - Gemini image analysis with circuit breaker and retry logic
  • db_service.py - Firestore operations with retry decorators for transient failures
  • storage_service.py - GCS file uploads with validation and retry logic
  • analytics_service.py - BigQuery queries with graceful degradation
  • geo_service.py - Geohashing and geocoding with circuit breaker for Nominatim
  • auth_service.py - JWT tokens, password hashing, OAuth integration

Error Handling Layer:

  • core/exceptions.py - 32 custom exception types (DatabaseError, AIServiceError, etc.)
  • core/error_handling.py - Circuit breakers, retry decorators, error context manager
  • core/validation.py - Input validation (files, coordinates, strings, reports)

Data Model:

  • Reports stored in Firestore reports collection with geohash indexing
  • Comments stored as subcollections under each report
  • Users collection with username-based authentication (migrated from email)
  • Timeline events array tracks state changes
  • Image hash (SHA256) stored for exact duplicate detection
  • Geohash field (precision 5) for proximity queries (~4.9km grid)

Key Implementation Details

Report Status Lifecycle:

  • PENDING_VERIFICATION → AI analysis in background
  • VERIFIED → Valid civic issue, no duplicates
  • REJECTED → AI found no valid civic issue
  • DUPLICATE → Similar issue exists nearby (same category, <55m radius)
  • IN_PROGRESS → Admin marked as being addressed
  • RESOLVED → Admin marked as completed
  • FLAGGED → System error during processing

Duplicate Detection Strategy:

  1. Exact duplicates: SHA256 hash comparison of uploaded images
  2. Proximity duplicates: Geohash + category matching within 0.0005 degrees (~55m)
  3. Checks only VERIFIED and PENDING_VERIFICATION reports from last 24h

Authentication:

  • JWT tokens with 30-day expiration (ACCESS_TOKEN_EXPIRE_MINUTES=43200)
  • Username-based authentication (supports phone, email, OAuth, or username login)
  • OAuth2 password flow at /api/v1/auth/token
  • Users have roles: “citizen” or “admin”
  • Admin endpoints use JWT authentication with get_current_admin() dependency

Geospatial Features:

  • Geohash encoding at precision 5 (~4.9km × 4.9km grid)
  • Firestore range queries: geohash >= prefix AND geohash < prefix~
  • Geocoding via Nominatim API (limited to India with countrycodes=in)
  • Keywords extracted from title+description for basic search

AI Analysis:

  • Gemini model called with image and structured JSON prompt
  • Returns: is_valid (bool), category (str), severity (1-10), description (str)
  • Circuit breaker prevents cascading failures (5 failures, 120s recovery)
  • Retry logic with exponential backoff (3 attempts, 4-60s)
  • Handles rate limiting (429) and quota exceeded errors
  • Falls back to FLAGGED status on persistent failures

BigQuery Analytics:

  • Table: darshi_analytics.reports_raw_changelog
  • Mimics Firestore changelog stream (document_name, timestamp, operation, data)
  • Hotspot query: severity >7, count >=3, within 24 hours

Important Configuration Notes

Google Cloud Setup Required:

  • Service account JSON must have permissions for:
    • Firestore read/write
    • Cloud Storage bucket creation and object uploads
    • BigQuery query execution
  • GCS bucket name format: {PROJECT_ID}-reports
  • Bucket location: ASIA-SOUTH1 (hardcoded)
  • Images made public via blob.make_public() (may fail if IAM restricted)

Security Considerations:

  • SECRET_KEY in production MUST be changed from development default
  • Rate limiting enforced on all public endpoints (tiered: anonymous 3/hr, registered 10/hr)
  • Input sanitization via sanitize_form_data() at router layer
  • File validation (max 10MB, image types only)
  • CORS configured per environment (allows localhost in development)
  • service-account.json should NEVER be committed (add to .gitignore)

Error Handling & Robustness

Custom Exception Hierarchy: All services raise specific exceptions instead of returning None/False:

  • DatabaseError → Firestore operations failures
  • StorageError → GCS upload/download failures
  • AIServiceError → Gemini API failures
  • GeocodingError → Geocoding/reverse geocoding failures
  • ValidationError → Input validation failures

Retry & Circuit Breaker Patterns:

  • Database operations: 3 retry attempts with 1-10s exponential backoff
  • AI operations: 3 retry attempts with 4-60s backoff
  • Storage operations: 2 retry attempts with 2-20s backoff
  • Circuit breakers on external APIs (Gemini: 5 failures/120s, Nominatim: 3 failures/60s)
  • Error context manager tracks operation duration and structured logging

Frontend Architecture

SvelteKit 5 Structure:

  • Uses new Runes system (state,derived, props,effect) for reactivity
  • No external CSS frameworks - all styling is custom and lightweight
  • TypeScript for type safety across all components
  • Centralized API client in lib/api.ts with structured error handling
  • Svelte stores for global state management (user, toast, etc.)

UX Components:

  • Toast.svelte - Non-blocking notifications (success/error/warning/info)
  • LoadingButton.svelte - Buttons with integrated loading spinners
  • LoadingSpinner.svelte - Reusable loading indicators (3 sizes)
  • lib/validation.ts - Client-side form validation utilities
  • All components follow WCAG 2.1 accessibility guidelines

Key Pages:

  • / (routes/+page.svelte) - Home page with reports listing and hotspot alerts
  • /submit (routes/submit/+page.svelte) - Report submission form with GPS and geocoding
  • /report/[id] (routes/report/[id]/+page.svelte) - Detailed report view with comments and timeline
  • /signin (routes/signin/+page.svelte) - User sign-in with OAuth, credentials, phone OTP, and email link
  • /register (routes/register/+page.svelte) - User registration
  • /admin (routes/admin/+page.svelte) - Admin dashboard with report management

State Management:

  • user store - Current authenticated user
  • isAuthenticated store - Authentication status
  • userLocation store - User’s GPS coordinates
  • toast store - Toast notification queue with auto-dismiss
  • localStorage for persistent auth token and user data

API Integration:

  • All API calls centralized in lib/api.ts
  • Structured error handling with getErrorMessage() utility
  • Parses backend error responses (code, message, details, recoverable flag)
  • Network error detection for offline scenarios
  • FormData for file uploads
  • Bearer token authentication
  • Type-safe interfaces for all API responses

UX Patterns:

  • Toast notifications instead of alert() dialogs
  • Loading states on all async operations
  • Optimistic UI updates with rollback on failure
  • Client-side form validation with inline errors
  • Character counters on text inputs
  • ARIA labels, semantic HTML, keyboard navigation

Responsive Design:

  • Touch-friendly buttons and forms

Logging Standards

Backend Logging Levels:

  • DEBUG: Internal operations, flow tracking (queries, lookups, coordinate encoding)
  • INFO: User-facing operations (report creation, user login, file uploads)
  • WARNING: Validation failures, recoverable errors (geocoding failures, rate limit approached)
  • ERROR: Actual failures with context (database errors, API failures, upload errors)
  • CRITICAL: System initialization failures (Firestore connection, Firebase Admin SDK)

Logging Best Practices:

  • Include relevant IDs in all log messages (report_id, user_id, request_id)
  • Use f-strings for log formatting: logger.info(f"Report {report_id} created by {username}")
  • Add exc_info=True to ERROR logs for stack traces
  • Never log sensitive data (passwords, tokens, PII)
  • No console.log statements in production frontend code

Code Quality Standards

Backend:

  • All services must raise specific exceptions (DatabaseError, AIServiceError, etc.)
  • Never return None/False for error conditions - raise exceptions with context
  • Use retry decorators for external operations (@retry_database_operation, @retry_ai_operation)
  • Add circuit breakers to external API calls (Gemini, Nominatim)
  • Validate all inputs at router layer before passing to services

Frontend:

  • Use toast notifications, never alert() dialogs
  • Add LoadingButton to all form submissions
  • Implement client-side validation with inline error messages
  • Ensure accessibility (ARIA labels, semantic HTML, keyboard navigation)
  • No console.log/console.error statements in production code
  • Use SvelteKit 5 Runes (state,derived, $effect) for reactivity
